# FulgurNET benchmark results

> Generated by `dotnet run -c Release --project FulgurNET.Benchmarks -- --full` on
> 2026-06-10 (UTC) on the developer's i5-1335U laptop.
>
> Rerun with the same command on your own machine — absolute numbers
> depend on CPU, GC tuning, and OS noise, but the relative ordering
> (FulgurNET > PuppeteerSharp > DinkToPdf) reproduces consistently.

## Methodology

- **Hardware:** 13th Gen Intel Core i5-1335U (10C/12T), 1 CPU, Server GC
- **.NET:** 10.0.6, X64 RyuJIT x86-64-v3
- **OS:** Windows 11 (10.0.26200)
- **Benchmark harness:** BenchmarkDotNet 0.15.3 with `ShortRun` (3 iterations, 1 launch, 3 warmup)
- **Diagnosers:** `MemoryDiagnoser` (managed allocation per op)
- **HTML fixtures:**
  - `Small` — ~3 KB, single-page table
  - `Medium` — ~12 KB, invoice with header, table, totals
  - `Large` — ~80 KB, 200 paragraphs of Lorem Ipsum
  - No JavaScript, no external assets (fair comparison: fulgur is
    pure HTML+CSS; Chromium would have an unfair edge on JS-driven pages)
- **Engine configuration:**
  - FulgurNET — direct P/Invoke to `fulgur_native.{dll,so,dylib}` via
    `csbindgen`-generated `NativeMethods.g.cs`. `Cold` constructs a new
    `FulgurPdf` per iteration; `Warm` reuses one.
  - DinkToPdf — `SynchronizedConverter` wrapping `PdfTools()`. `Cold`
    rebuilds the converter per iteration; `Warm` reuses one. Requires
    `libwkhtmltox` on `PATH` (auto-staged by `DinkToPdfBench.EnsureLibWkhtmltoxPresent`).
  - PuppeteerSharp — single Chromium browser launched in `[GlobalSetup]`,
    reused across iterations. `--no-sandbox` to support Windows containers.

## Caveats

- **BDN `ShortRun` has high variance on warmup-sensitive engines.** A
  158× speedup is enormous; even with 50% noise, the ordering is robust.
- **Cold vs warm.** FulgurNET has near-zero cold-start cost (one FFI
  call into the Rust engine) — the "warm" speedup is mostly Puppeteer
  paying its Chromium IPC tax. DinkToPdf pays a per-call PDF rendering
  tax whether or not the converter is reused.
- **DinkToPdf's allocations are smaller but its latency is 100× worse.**
  This is because wkhtmltopdf 0.12.6 (Qt-patched) does most of its
  work in a subprocess with heavy string conversion overhead.
- **FulgurNET allocations** are 100% managed byte[] — there are zero
  unmanaged leaks in `FulgurPdf.FromHtml`'s contract.

## Reproduce

```sh
# Quick smoke (one render per engine, no BDN, ~10 s):
dotnet run -c Release --project FulgurNET.Benchmarks

# Full BDN run with competition report (~3 min including Chromium warmup):
dotnet run -c Release --project FulgurNET.Benchmarks -- --full
```

The full run ends with the same headline table printed to stdout, plus
per-size detail tables.
